# 77. DHCP (Dynamic Host Configuration Protocol)

### **1. Назначение и цель**

**DHCP (Dynamic Host Configuration Protocol)** — это **прикладной протокол** (уровень 7 модели OSI, уровень Application в TCP/IP), предназначенный для **автоматической выдачи сетевых параметров** хостам при подключении к сети.

**Основная задача**: устранить необходимость **ручной настройки IP-адресов**, масок подсети, шлюзов и других параметров, что особенно важно в крупных сетях.

---

### **2. Какие параметры выдаёт DHCP?**

DHCP-сервер может предоставить клиенту следующие параметры:
- **IP-адрес** (основной),
- **Маска подсети**,
- **IP-адрес шлюза по умолчанию**,
- **IP-адреса DNS-серверов**,
- **Имя домена**,
- **Адресы серверов времени (NTP)**,
- **Имя загрузочного файла** (для PXE-загрузки),
- **Срок аренды IP-адреса (lease time)**.

> 💡 Без DHCP в крупной сети администратору пришлось бы вручную настраивать сотни/тысячи устройств.

---

### **3. Работа DHCP: DORA-процесс**

Выдача параметров происходит в **четыре этапа**, известных как **DORA**:

#### **1. DISCOVER (обнаружение)**
- Клиент включается **без IP-адреса**.
- Отправляет **широковещательный запрос** (`DHCPDISCOVER`) по MAC-адресу `ff:ff:ff:ff:ff:ff`.
- Порт назначения: **UDP 67**.

#### **2. OFFER (предложение)**
- Все DHCP-серверы в сети получают DISCOVER.
- Каждый сервер **резервирует IP-адрес** и отправляет **unicast или broadcast-ответ** (`DHCPOFFER`) с предложением параметров.
- Клиент может получить несколько OFFER.

#### **3. REQUEST (запрос)**
- Клиент **выбирает один OFFER** (обычно первый полученный).
- Отправляет **широковещательный запрос** (`DHCPREQUEST`), указывая, **какой сервер выбран**.
- Другие серверы **освобождают зарезервированные адреса**.

#### **4. ACK (подтверждение)**
- Выбранный сервер подтверждает выдачу (`DHCPACK`).
- Клиент **принимает параметры** и может использовать сеть.
- Если сервер не может выдать адрес — отправляется `DHCPNAK`.

> ⚠️ Все сообщения DHCP передаются поверх **UDP**:
> - Клиент → сервер: **UDP 68 → UDP 67**
> - Сервер → клиент: **UDP 67 → UDP 68**

---

### **4. Структура DHCP-сообщения**

DHCP использует тот же формат, что и **BOOTP** (его предшественник). Основные поля:

| Поле | Описание |
|------|--------|
| **op** | Тип: `1` = запрос (client), `2` = ответ (server) |
| **htype, hlen** | Тип аппаратного адреса (Ethernet = 1), длина (6 байт для MAC) |
| **hops** | Число ретрансляций (для DHCP-реле) |
| **xid** | Идентификатор транзакции (для сопоставления запрос/ответ) |
| **yiaddr** | Предложенный IP-адрес (в OFFER и ACK) |
| **siaddr** | IP-адрес сервера |
| **giaddr** | IP-адрес DHCP-релея (если используется) |
| **chaddr** | MAC-адрес клиента |
| **options** | Основные параметры: `53` = тип сообщения, `51` = срок аренды, `54` = IP сервера, `50` = запрошенный IP и др. |

---

### **5. Аренда IP-адреса (Lease)**

- IP-адрес выдаётся **временно** — на **срок аренды (lease time)**, например, 24 часа.
- Клиент **автоматически продлевает аренду**:
  - В **50% срока** → отправляет **unicast REQUEST** серверу.
  - Если нет ответа — в **87.5% срока** → отправляет **broadcast REQUEST**.
- По истечении срока без продления адрес **возвращается в пул**.

> 💡 Это позволяет **эффективно использовать ограниченный пул IP-адресов** (например, в гостиницах, Wi-Fi кафе).

---

### **6. DHCP Relay (ретранслятор)**

- По умолчанию DHCP работает **только в одной подсети**, так как DISCOVER — широковещательный пакет.
- **DHCP Relay Agent** (обычно на маршрутизаторе) перехватывает broadcast-запрос и **пересылает его unicast** на DHCP-сервер в другой сети.
- При этом заполняет поле **giaddr** (gateway IP address) — IP интерфейса, с которого пришёл запрос.

---

### **7. Безопасность и ограничения**

#### **Уязвимости:**
- **DHCP Spoofing** — злоумышленник запускает поддельный DHCP-сервер и выдаёт свои адреса (для MITM-атак).
- **DHCP Starvation** — атакующий запрашивает все адреса из пула → DoS для новых клиентов.

#### **Меры защиты:**
- **DHCP Snooping** (на коммутаторах) — фильтрация DHCP-сообщений только от доверенных портов.
- **Резервирование адресов** по MAC-адресу для критических устройств.
- Использование **статических адресов** для серверов.

---

### **8. DHCP в IPv6**

В IPv6 используется **DHCPv6**, но также доступна **бесконфигурационная настройка (SLAAC)** через **ICMPv6 Router Advertisements**.

- **Stateless DHCPv6** — выдаёт только параметры (DNS, NTP), но не IP-адрес.
- **Stateful DHCPv6** — выдаёт и IP-адрес, и параметры.

---

### **9. Команды для работы с DHCP (Linux/Windows)**

| ОС      | Команда                                     | Действие                       |
| ------- | ------------------------------------------- | ------------------------------ |
| Linux   | `dhclient eth0`                             | Запросить адрес для интерфейса |
| Linux   | `systemctl restart NetworkManager`          | Перезапустить сетевой менеджер |
| Windows | `ipconfig /release`                         | Освободить текущий адрес       |
| Windows | `ipconfig /renew`                           | Запросить новый адрес          |
| Обе     | `journalctl -u dhclient` или `Event Viewer` | Просмотр логов DHCP            |

---

### **10. Заключение**

DHCP — это **критически важный протокол** современных сетей, обеспечивающий:
- **автоматизацию** настройки сетевых параметров,
- **гибкое управление** пулом IP-адресов,
- **упрощение администрирования** в сетях любого масштаба.

Без DHCP развертывание и поддержка даже небольшой корпоративной сети стала бы крайне трудоёмкой задачей.

> 📌 **Примечание**: В методических указаниях по БЭВМ-NG **DHCP не упоминается**, так как БЭВМ — учебная модель процессора без сетевых функций.
