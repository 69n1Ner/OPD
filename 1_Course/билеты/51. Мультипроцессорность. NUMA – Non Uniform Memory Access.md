### **51. Мультипроцессорность: NUMA (Non-Uniform Memory Access)**
**NUMA** — архитектура мультипроцессорной системы, в которой **время доступа к памяти зависит от её физического расположения** относительно процессора.
#### **Основные черты**
- Система делится на **узлы (nodes)**, каждый содержит CPU(ы) и локальную память.
- Доступ к **локальной памяти** быстрее, чем к **удалённой** (в 1.5–3 раза).
- Единое логическое адресное пространство.
- Поддержка **аппаратной когерентности кэшей** через интерконнект.
#### **Преимущества**
- Высокая масштабируемость (десятки–сотни ядер),
- Эффективность при хорошей локальности данных.
#### **Недостатки**
- Сложность программирования: производительность зависит от размещения данных,
- Потери при частом доступе к удалённой памяти,
- Требует **NUMA-aware ОС** (Linux, Windows Server).
#### **Примеры**
- **Intel Xeon Scalable** (интерконнект UPI),
- **AMD EPYC** (Infinity Fabric, до 8 NUMA-узлов на одном чипе).
#### **Сравнение с UMA**
| Параметр | UMA | NUMA |
|---------|-----|------|
| Время доступа | Равномерное | Зависит от расстояния |
| Масштабируемость | До 8–16 ядер | До сотен ядер |
| Простота программирования | Высокая | Требует учёта локальности |

# 51. Мультипроцессорность: NUMA – Non Uniform Memory Access

### **1. Определение**

**NUMA (Non-Uniform Memory Access)** — это архитектура мультипроцессорной системы, в которой **время доступа к памяти зависит от физического расположения** запрашиваемых данных относительно процессора.

> 💡 Ключевая идея: у каждого процессора (или группы ядер) есть **«своя» локальная память**, а доступ к «чужой» (удалённой) памяти занимает **больше времени**.

---

### **2. Структура NUMA-системы**

```
   +----------------+        +----------------+
   | Узел 0         |        | Узел 1         |
   |                |        |                |
   |  CPU 0, CPU 1  |<──────>|  CPU 2, CPU 3  |
   |  Локальная     |        |  Локальная     |
   |  память (RAM0) |        |  память (RAM1) |
   +----------------+        +----------------+
            \                      /
             \                    /
              \                  /
               \                /
                \              /
                 \            /
                  \          /
                   \        /
                    \      /
                     \    /
                      \  /
                   +----------+
                   | Интерконнект |
                   | (QPI, UPI,  |
                   |  Infinity   |
                   |  Fabric)    |
                   +----------+
```

- Система делится на **узлы (nodes)**,
- Каждый узел содержит **один или несколько процессоров и банк ОЗУ**,
- Узлы соединены **высокоскоростным интерконнектом**.

---

### **3. Основные характеристики**

| Характеристика | Описание |
|----------------|---------|
| **Неравномерный доступ** | Доступ к локальной памяти быстрее, чем к удалённой (обычно в 1.5–3 раза) |
| **Единое адресное пространство** | Программа видит всю память как одну непрерывную область |
| **Когерентность кэша** | Поддерживается на аппаратном уровне (через протоколы и интерконнект) |
| **Масштабируемость** | Высокая — поддерживает десятки и сотни процессоров |

---

### **4. Преимущества NUMA**

- **Высокая масштабируемость**:  
  Нет единой шины или коммутатора, ограничивающего рост.
- **Эффективность при локальных данных**:  
  Если поток работает с данными в своей локальной памяти — производительность максимальна.
- **Гибкость конфигурации**:  
  Можно строить системы с разным числом узлов и объёмом памяти.

---

### **5. Недостатки**

- **Сложность программирования**:  
  Производительность сильно зависит от **размещения данных**.  
  Пример: если поток на CPU 0 постоянно читает данные из RAM1 — система тормозит.
- **Потеря производительности при «плохой» локальности**:  
  Удалённый доступ может быть в 2–3 раза медленнее.
- **Требования к ОС**:  
  Необходима **NUMA-aware** операционная система, которая:
  - Размещает потоки ближе к их данным,
  - Выделяет память локально,
  - Балансирует нагрузку между узлами.

---

### **6. Примеры реализации**

| Платформа | Интерконнект | Особенности |
|----------|--------------|------------|
| **Intel Xeon Scalable** | UPI (Ultra Path Interconnect) | До 8 сокетов, каждый с локальной DDR4/DDR5 |
| **AMD EPYC** | Infinity Fabric | До 8 сокетов, 8 NUMA-узлов на одном чипе (в Milan/Rome) |
| **SGI Altix, HP Superdome** | Proprietary interconnects | Исторические крупные NUMA-системы |

> 💡 Даже **одночиповые серверы AMD EPYC** с 64 ядрами являются **многоузловыми NUMA-системами** (до 8 узлов на кристалле).

---

### **7. NUMA vs UMA**

| Параметр | **UMA** | **NUMA** |
|---------|--------|---------|
| Доступ к памяти | Равномерный | Зависит от расстояния |
| Масштабируемость | До 8–16 ядер | До сотен ядер |
| Простота программирования | Высокая | Требует NUMA-aware кода |
| Производительность при локальных данных | Хорошая | Отличная |
| Производительность при удалённых данных | Хорошая | Плохая |

---

### **8. Поддержка в операционных системах**

- **Linux**:  
  - Команды: `numactl`, `lscpu`, `/sys/devices/system/node/`,  
  - Политики размещения: `localalloc`, `interleave`, `preferred`.
- **Windows**:  
  - NUMA-aware планировщик,  
  - API: `GetNumaNodeProcessorMask`, `VirtualAllocExNuma`.

> 💡 Без NUMA-aware ОС преимущества архитектуры теряются.

---

### **9. Заключение**

NUMA — это **естественная эволюция мультипроцессорных систем**, позволяющая преодолеть ограничения UMA при большом числе процессоров. Она обеспечивает **высокую масштабируемость**, но требует **осознанного подхода к размещению данных и потоков**. В современных серверах и HPC-системах NUMA стала **стандартом де-факто**, а понимание её принципов необходимо для написания высокопроизводительного кода.
